## 用法 / 分析深度

**定位**：本表是**需求澄清 / 待决议范围清单**（open decision log），不是最终设计方案，也不是周任务看板。议题保持开放；拍板后填「决议」并改状态为 `closed`；新增议题追加行即可。**不要**因单条口头意见直接改 `requirements.md` 硬范围。

**默认用决策卡，不要逐项 SWOT**：OD-1～6 多为工程权衡，逐条做完整 SWOT 过重，且与张老师原则（目的：准确度/效率/成本 > 手段复杂度）不符。每项用决策卡字段推进：

| 字段 | 含义 |
|------|------|
| 选项 | 可选方案（可先空，会上补） |
| 三轴影响 | 对**准确度 / 效率 / 成本**的影响（可写「待测」） |
| 主要风险 | 技术、进度、M币、耦合风险 |
| 建议与前置 | 倾向方案 + 需先确认的依赖 |
| 状态 / 决议 | 见下表状态约定 |

**仅对高耦合、改变系统形态的分叉**做轻量方案对比（可称 mini-SWOT / 方案对比），例如：

- **OD-4.3**（及与 **OD-1.1** 的耦合）：连续输送 + 全局快门 vs 启停 + 卷帘  
- **OD-3.2**：无监督拓展是否进本期范围  

其余（预处理是否需要、中期 PPT、节点细化等）填决策卡即可，不上完整 SWOT。

## 状态约定

| 状态 | 含义 |
|------|------|
| `open` | 待沟通 / 待决议 |
| `discussing` | 已上会、未拍板 |
| `closed` | 已决议（须填决议摘要与日期） |
| `deferred` | 明确延期 |

## 如何更新（给后续同步）

1. 改对应行的「状态 / 决议 / 更新日期」。  
2. 重要决议另写会议纪要：`docs/05-reporting/meetings/YYYY-MM-DD-*.md`，并在决议栏链到纪要。  
3. 跑 `apps/bi` 下 `npm run sync-data`，BI 知识库即可看到新版本。  
4. **不要**因单条口头意见直接改 `requirements.md` 硬范围；范围变更走变更控制。
