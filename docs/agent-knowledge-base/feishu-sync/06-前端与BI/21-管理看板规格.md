2. 风险页（登记册热力表）
目的：服务每周风险评审与触发条件监控（WBS 6.1/6.2）：哪些风险该升级、谁是 owner、有没有超期未更新（FR-P2）。
数据字段（建议 .project-spec/risks.json，登记册正文见《05-项目管理/15·风险登记册》）：id（R1~R7 起）/ title / owner / probability(1-5) / impact(1-5) / score(=P×I) / trigger（触发条件）/ mitigation / status（open / mitigating / closed）/ updated_date。
热力表（建议）：5×5「概率 × 影响」网格，格内放风险 ID 徽标；分档建议：score ≥15 红、8~14 黄、≤7 绿；点徽标弹出风险卡（触发条件 + 应对 + owner + 最近更新）。
现状与落地：L0 总览已有风险区（已有）；独立热力页为建议（不新增路由时可作为 L0 风险区的下钻展开）。
灰显规则：登记册为空 → 整区灰显「无已登记风险」；缺 owner 或超 14 天未更新的风险徽标加「!」——视为数据质量问题高亮，而非静默。

5. 知识库页（/knowledge，已有）
目的：服务队员 48h 上手与免登录知识获取；游客与校友导师也可读（免登录是已有机制）。
同步机制（已决议）：策展文档 + 意见箱 inbox/ 与 digests/ → npm run sync-data → public/knowledge/ + 生成 public/data/knowledge-index.json；手动触发，改文档后须重跑。
数据字段（knowledge-index.json）：id / title / type（领域知识/会议纪要/ digest 等）/ updated / tags / path。
页面建议：置顶「入门路径」（onboarding → 评价原则 → 缺陷图谱）；按 type 分组列表；搜索框（前端本地索引，无后端）。知识库内容同时承担「对外叙事」职能：导师与第三方通过它理解项目口径，因此条目标题与摘要须用中文业务语言，避免只有内部缩写；过期文档不删除，标「已过期」灰显归档，保留演进痕迹。
灰显规则：索引为空 → 整页灰显「知识库待同步」；单篇文件缺失 → 条目灰显且不可点，不抛 404 页。
