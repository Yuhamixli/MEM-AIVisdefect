2. 逐项论证
2.1 前端框架：React + Vite + TS（延续）
bi 已用 React/Vite/TS 跑通五个页面（/ /budget /team /knowledge /feedback），构建、代理、Pages 部署全链路验证过。程昱涵作为主前端沿用同一栈，detector-ui 直接复制工程骨架。
三轴影响：准确度——同栈意味着缺陷数据的 slug/时间等口径在两应用间天然一致，减少「两套框架两种格式」的对齐错误；效率——零迁移成本，中期前的人时全花在业务页而不是学新框架；成本——不新增依赖许可证与体积开销，M币人时趋近于复用。
2.2 UI 库：Tailwind + 轻组件，不引重框架
BI 的设计基调已定（工业纸感底 + 松绿强调 + Newsreader/IBM Plex，明确不做紫白渐变模板）。引一套重组件库（成套 admin 模板）会把视觉风格拖回默认皮肤，反而要花人时覆盖样式。建议 Tailwind 写原子样式 + 少量自绘组件（表格、标签、卡片、弹层），组件总数控制在 15 个以内。
三轴影响：准确度——视觉规范单点维护，数值/状态色不会因库默认主题串味（如风险红的色值统一）；效率——无库版本升级与 breaking change 负担，湛佳谕测前端时定位样式问题路径短；成本——CSS 产物为按需裁剪的极小体积，Pages 免费额度内秒开，不花一分钱 CDN。
2.3 仓结构：detector-ui 落在 apps/ 下同仓新包
仓库地图本就预留 apps/ bi/（管理看板）· detector-ui（待建）。两应用同仓：共享 .project-spec 数据契约、共享 sync-data 思路、共享 deploy 工作流模板。
三轴影响：准确度——检测 schema（文档 22 组三）与 BI 契约同仓评审、同 PR 演进，接口漂移当天可见；效率——一次 clone 即可联调，黄崇发算法输出的样例 JSON 直接放 apps/detector-ui/public/data/detect/ 供前端 mock；成本——不新增仓库、不设第二套权限与 CI 密钥。
2.4 图像叠加：canvas 自绘，不引重标注库
detector-ui 核心画面 = 一张检测图 + 若干 [x,y,w,h] 框 + 类别/置信度标签。这是只读叠加，canvas 自绘一个 <BoxOverlay> 组件（约百行：图片适配缩放、坐标换算、悬停高亮、按 slug 配色）即可覆盖。标注编辑类库（带拖拽改框、多边形、快捷键体系的那种）体积大、学习曲线陡，且其能力 90% 本期用不到——标注生产在 LabelImg 侧（data 工作流），不在 BI。
回退触发条件（写死）：若 /api/review 复核需求从「确认/拒绝」升级为「在线改框/重标类别」（见文档 22 组三草稿），且 8 月复核量实测 >50 框/天，再评估引入标注库；在此之前不预引。
三轴影响：准确度——自绘换算逻辑只有一处（原图像素↔显示坐标），框不漂移；效率——不啃第三方库 API，联调检查单（文档 26）当天可验证；成本——零新增依赖，页面加载体积≈图片本身，产线旧电脑浏览器也能跑。
2.5 部署：静态构建 + Cloudflare Pages（主），内网静态托管（备）
两应用都是纯静态产物 + 少量 Pages Functions（意见箱、健康检查）。主路径延续现状：push 到 main → GitHub Actions 构建 → Pages 发布。备选：把 dist 打成 tar 包，内网 nginx 静态托管（应对评审现场无外网或 V公司内网演示场景），Functions 不可用时的回退见 §3。
三轴影响：准确度——Pages 每次部署对应一个 commit，演示版本与仓库状态严格一致，可追溯；效率——push 即发布，免运维，评审前出包时间从小时级降到分钟级；成本——Pages 免费额度覆盖本期全部流量，内网备用方案零增量采购（复用现有机器），总成本≈0 元 + 少量 M币人时。
2.6 状态管理：URL 状态 + fetch 缓存，不引 Redux
页面级状态（当前路由、筛选条件、选中样件）放 URL query，天然可分享、可刷新还原；服务端数据用带缓存的 fetch 封装（内存 Map + SWR 策略：先给缓存、后台重取）。全局客户端状态几乎没有（无登录态、无购物车式交互），Redux/MobX 级别的方案是杀鸡用牛刀。
三轴影响：准确度——URL 即状态快照，bug 复现只需一个链接，评审演示可直接深链到某样件的某个面；效率——模板代码趋近于零，新页面半天起步；成本——依赖少一层，构建与排错都便宜。
2.7 构建与 CI：deploy-bi.yml 现状延续
现状流水线：sync-data → build → pages deploy（secrets：CLOUDFLARE_API_TOKEN / CLOUDFLARE_ACCOUNT_ID）。建议只做两处增量，不改骨架：① detector-ui 建好后加一条并行 job 复用同一模式；② PR 上加 tsc --noEmit + vite build 的校验 job（不部署），防破窗。
三轴影响：准确度——每次合并前有类型与构建双闸门，坏构建不上一线；效率——流水线已跑通，增量改动半天内完成；成本——GitHub Actions 公开/私有仓免费额度内，零采购。

4. 与三轴的一句话总结
这套选型没有一个「新技术」，全部决定都指向同一件事：把有限的 M币人时花到检测准确度证据链（50 件评测、结构化输出、复核闭环）上，而不是花到框架本身——前端对课题的贡献是让数据可信地被看见，这是它独有的准确度价值。
