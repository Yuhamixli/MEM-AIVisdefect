目的：给「意见与需求收集」定一条本期可执行、可追溯、不扰动范围基线的通道。铁律：意见箱只收集与分流，不因单条意见自动改需求/章程（对齐 change-control 与 feedback 工作流职责）；周度 digest 在周五前完成，digest 只进知识库与例会议题，不进章程正文。
1. 两个候选方案
方案 A：网页表单 → Pages Function → GitHub 写回（现状延续）
链路：BI /feedback 页 → 口令解锁 → POST /api/feedback（Cloudflare Pages Function，校验 WRITE_PASSWORD）→ GitHub contents API 写入 docs/feedback-inbox/inbox/YYYY-MM-DD-<slug>.md → 周五前人工 digest → digests/ 归档、原件移 processed/。
现状：本地 dev:api（8788）已可跑通（默认口令 dev-password 写本地 inbox/）；生产只差在 Cloudflare 配置 secrets（WRITE_PASSWORD / GITHUB_TOKEN / GITHUB_REPO / GITHUB_BRANCH），属「已验证、待公布 URL」状态。
特点：意见与需求同仓留痕，天然带 git 历史与 blame；与知识库、未决清单同一信息源；队员看知识免登录，填意见仅需队入口令。
代价：依赖 GitHub API 可用性与 token 保管；非技术填写人看不见「落盘」动作，需要页面回执增强信任。
方案 B：网页表单 → 飞书多维表格
链路：BI /feedback 页内嵌（或跳转）飞书多维表格「表单视图」→ 填写即入表 → 自动化提醒到飞书群 → 每周由 owner 导出/整理为 md 归档到仓库 docs/feedback-inbox/digests/。
特点：零后端维护；填写体验对非技术成员友好；可勾选匿名；多维表格自带状态字段（待认领/处理中/已闭环），看板视图直接当周会材料；提醒到群时效好。
代价：数据先在飞书、后在仓库，存在两个事实源，导出归档靠人肉，漏导即断链；飞书租户与仓库的权限口径需要单独维护；按 D-06 飞书可放项目文档，但意见原文可能含敏感吐槽，归档进 git 前需再过一道脱敏。
2. 对比（按准确度/效率/成本三轴 + 可维护性）


维度

A：GitHub 写回

B：飞书多维表格

准确度（信息保真/可追溯）

高：与仓库同源，git 历史即审计链，digest 引用 commit 可回溯原文

中：双事实源，导出环节有遗漏/改写风险

效率（填写→被看见的时延）

中：每周五 digest 后集中可见；单条不推送

高：提交即提醒到群，适合需要快响应的收集

成本（建设+维护，含 M币人时）

低：函数已写好，维护≈0；secrets 一次性配置

中：多维表格+自动化要搭，且每周人肉导出是长期人时

填写门槛

需打开 BI 站点+口令

飞书原生，手机端友好，可匿名勾选

权限/隐私

仓库权限即边界；原文入 git 后全员可见

表格权限单独管；敏感内容留飞书可圈定可见范围

失效回退

GitHub 挂→本地导出再入仓（见文档 24 风险节）

飞书挂→表单不可用，无本地缓存
3. 推荐结论（建议，待范汝杰确认）
A 为主通道，B 为专项备选。 理由三句：① A 已跑通且与仓库同源，准确度（可追溯性）最好，长期维护成本最低；② 本期意见主体是组内成员，口令门槛可接受；③ B 的「提醒到群+手机端」优势在两类场景确实更好——校友导师（朱定涛）意见收集与课程侧材料/问卷收集，这两类不进仓库也无妨、且要求低门槛，建议用 B 承接，每周由 feedback 兼岗把有效条目摘要进 digest（只摘结论，不搬原文，敏感内容留飞书）。
即：组内意见=A（唯一入口，避免双通道分裂）；对校友导师/课程材料的收集=B。两通道的 digest 出口合一，都在 docs/feedback-inbox/digests/。
4. 完整流程图（文字泳道）
泳道        步骤1        步骤2        步骤3         步骤4          步骤5           步骤6
[填写人]   打开/feedback ─→ 输口令解锁 ─→ 填表单提交 ─→ 看到回执(file路径)                              ─→ 周会看到digest
[前端]     渲染表单      ─→ POST /api/feedback ─→ 200显示回执/4xx红标 ─→ 失败保留草稿(localStorage)
[Function]              ─→ 校验口令(401) ─→ 校验字段(422) ─→ slug化标题 ─→ 调GitHub contents API
[GitHub]                                                     ─→ inbox/2026-07-18-xxx.md (commit)
[owner认领]                                                          ─→ feedback兼岗每日扫inbox/
                                                                       ─→ 按category指派owner
                                                                       ─→ 条目转「未决清单」或「TODO」
[digest]                                                                ─→ 周五前汇总写digests/2026-W29.md
[归档]                                                                   ─→ 原件移processed/ ─→ 条目闭环
关键规则：① 口令错/字段缺的失败反馈不吞掉已填内容；② owner 认领只做「分流」（未决清单/TODO/知识库修订），不直接改章程与需求基线，改范围走变更控制；③ digest 之前条目状态保持 open，任何人可在 inbox/ 原文上追加评论式补充（直接编辑 md 或再提一条关联意见）。

6. 隐私与脱敏约定
不收集真名：表单 contact 字段为可选项，留空即匿名；只在「需要回访澄清」时建议填写。飞书 B 通道同理开启「允许匿名提交」。
默认脱敏：digest 引用意见时按 slug 化标题+日期引用，不引用联系方式；含个人信息、客户现场敏感信息（V公司人名/产线编号/价格）的原文在移 processed/ 前做一道替换（[姓名]/[产线]），保留语义。
权限边界：仓库私有为前提（对齐 NFR-4：企业/自有数据默认不公开）；B 通道表格仅课题组+校友导师可见；两通道都不截图外传，宣传材料须按课程规则先过执行组老师审核。
口令卫生：生产口令仅存 Cloudflare secrets，不入 git、不写进文档；本地 dev-password 仅开发用；怀疑泄露即轮换口令并同步到队内私密渠道。
