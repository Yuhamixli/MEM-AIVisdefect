1. 设计原则与总体架构
三条原则：① 验收主线最短化——任务书 P0（≥3 类、召回≥80%、准确率≥85%、50 件）由离线检测模块一线承载，其余能力不得挡主线；② 部署形态修正（ADR-001，已决议）——四层能力栈采纳为愿景，但禁止串成产线每帧主路径（串行召回相乘，两层各 90%→81%，对 80% 门槛无余量），改为「在线轻量 + 近线精修 + 离线飞轮」；③ 受控成像先于算法——光源不凭经验锁定，先 DOE 再冻结。

1.2 架构要点解读
采集子系统是准确度的第一来源：缺陷尺度 8~10mm、精度要求 5mm（加严），像元精度由分辨率二选一与物距共同决定（OD-COM-16）；反光与低对比缺陷靠光源 DOE 解决，不凭经验锁光源——成像不定，后层一切指标不可复现。
在线 L1 只跑轻量：PatchCore 只需良品即可训练，契合「暂无标注数据」的冷启动；蒸馏 YOLO 为可选项，是否常驻以节拍实测为准。
近线 L2/L3 是工具层不是判决层：DINO 擅长少样本定位但推理偏重，SAM2 掩码利于面积/严重度定量，二者服务复核与精修，不承担 P0 指标。
离线模块是验收主体：yolo11s-seg 环境已就绪，实例分割权重可退化输出纯检测框，CPU 可跑，第三方按使用说明即可独立操作（G6/G7）。
飞轮 L4 放大标注产能：VLM 预标注 + 人工确认，解决 13 人团队标注瓶颈；产出训练集版本反哺模型迭代，形成数据闭环。
展示层一屏两角：detector-ui 面向「检测结果」（框/列表/回放/复核），BI 看板面向「项目管理」（节点/风险/M币/知识库/意见箱），共用同一仓库数据源（npm run sync-data），不做双份事实源。

3. 数据流设计
raw-data 版本化入库
  → 标注（LabelImg/X-AnyLabeling；train/val/test = 7:2:1；抽检 10%；边界样本双人复核）
    → 训练/验证/测试严格隔离（按 physical_piece_id/批次拆分，防数据泄漏）
      → 50 件盲测（盲测冻结；报逐类指标 + Wilson 置信区间——
         例：40/50 的 95% Wilson 区间约 67%~88.8%，50 件只够过门禁、不足以证量产）
        → 结构化输出 JSON（件-批-面-时-模型版本最小追溯链）
          → SPC/批次追溯叙事（批次级缺陷率、分类分布进 BI 看板）
要点：① 数据版本与实验记录绑定（NFR-2），换模型版本须重跑基线批保证可比；② 评测可信度优先——按件拆分、盲测冻结、报置信区间，优于报单点高指标；③ 复核确认样本回流训练集，经版本升级反哺模型（L4 飞轮出口）；④ 标注侧工具现状为 LabelImg，升级 X-AnyLabeling 后可直接承接 VLM 预标注结果做人工修正，使飞轮在工具链上闭环（建议项，待 data 工作流确认）。

5. 关键选型论证（每项一句三轴论证）


选型

三轴论证（准确度/效率/成本）

PatchCore（L1）

良品即可训练、零缺陷样本也能先跑筛查——效率与成本最优；但只答「异不异常」，准确度上限靠后层与人工复核补。

Grounding DINO（L2，过渡）

文本提示零样本定位、标注少时效率最高；但推理重、难过节拍——放近线提效，主检测器达标即退场。

SAM2（L3，按需）

不规则缺陷掩码利于面积/严重度定量（准确度叙事加分）；延迟与算力成本高——只跑难例，不进每帧主路径。

yolo11s-seg（离线主检测）

环境已就绪、CPU 可跑、分割可退化出框——以最低成本直接承载 P0 指标（召回/准确率）；最终定版待 OD-COM-07 三轴实测。

VLM（L4）

预标注把标注产能放大数倍（效率/成本）；有幻觉风险——人工确认 + 10% 抽检兜底保准确度。

卷帘面阵+走-停-走

相机与机械改造成本最低、本期够用；运动模糊风险用「停拍」规避——全局快门+连续输送入量产路线图（OD-COM-03）。

React/Vite + Cloudflare Pages

延续既有仓库资产、零后端运维成本；前端人时全部投向展示与复核效率。
